# FeatherFTP 客户端实验原理与实验方案

## 1. 文档目标

本文档用于配合计算机网络实践课程实验报告，系统说明 FeatherFTP 图形化 FTP 客户端的实验原理、总体方案、协议实现、功能设计和测试验证思路。

本项目从零实现一个可在 Windows 环境运行的图形化 FTP 客户端。客户端底层 FTP 交互全部基于 Python 标准库 `socket` 手写实现，没有使用 `ftplib` 或其他高级 FTP 客户端库。系统提供类似 WinSCP 的双栏文件管理界面，支持连接管理、站点管理、目录浏览、上传、下载、断点续传、暂停继续、拖拽传输、右键文件操作、复制剪切粘贴、协议日志和传输队列可视化等功能。

## 2. 实验原理

### 2.1 FTP 协议概述

FTP 全称 File Transfer Protocol，即文件传输协议，是应用层协议之一，主要用于在客户端和服务器之间进行文件上传、下载和远程文件系统管理。FTP 建立在 TCP 之上，依赖 TCP 提供可靠、有序、面向连接的字节流传输能力。

FTP 的一个重要特征是采用控制连接与数据连接分离的设计：

- 控制连接：用于传输命令和响应，例如 `USER`、`PASS`、`PWD`、`CWD`、`LIST`、`RETR`、`STOR` 等。控制连接通常连接到服务器的 21 端口，也可以使用自定义端口，例如本项目 demo 服务器使用 2121 端口。
- 数据连接：用于实际传输目录列表或文件内容。每次执行 `LIST`、`MLSD`、`RETR`、`STOR` 等涉及数据传输的命令时，通常都会额外建立一条数据连接。

这种双连接设计使 FTP 的命令控制和大块数据传输相互独立。控制连接保持会话状态，数据连接按需创建和关闭。

```text
客户端控制 socket  <========== FTP 命令/响应 ==========>  服务器控制端口
客户端数据 socket  <========== 文件/目录数据 ==========>  服务器临时数据端口
```

### 2.2 TCP 与 Socket 编程基础

Socket 是操作系统提供给应用程序的网络编程接口。对于 TCP 客户端而言，典型过程如下：

1. 创建 socket。
2. 调用 `connect(host, port)` 与服务器建立 TCP 三次握手。
3. 通过 `send` 或 `sendall` 发送字节数据。
4. 通过 `recv` 或文件式 reader 读取服务器数据。
5. 使用 `close` 释放连接资源。

FTP 客户端在控制连接上发送文本命令。每条 FTP 命令通常以 `\r\n` 结尾，例如：

```text
USER demo\r\n
PASS demo123\r\n
TYPE I\r\n
PWD\r\n
```

服务器响应由三位数字状态码和文本说明组成，例如：

```text
220 FeatherFTP writable demo server ready.
331 Username ok, send password.
230 Login successful.
257 "/" is the current directory.
```

本项目的 `FtpSession` 类直接使用 `socket.create_connection()` 建立 TCP 控制连接，并用 `makefile("rb")` 以文件流方式读取 FTP 响应。所有命令由 `_send_command()` 组装并发送，所有响应由 `_read_reply()` 解析。

### 2.3 FTP 响应码机制

FTP 服务器响应以三位数字码开头。客户端需要根据响应码判断命令是否成功、是否需要继续发送命令、是否正在开始数据传输。

常见响应码如下：

| 响应码 | 含义 | 示例场景 |
| --- | --- | --- |
| 220 | 服务就绪 | 控制连接建立后的欢迎信息 |
| 230 | 用户登录成功 | `USER/PASS` 成功后 |
| 331 | 用户名正确，需要密码 | `USER` 后需要继续发 `PASS` |
| 150 | 文件状态正常，即将打开数据连接 | `LIST/RETR/STOR` 开始前 |
| 200 | 命令成功 | `TYPE I` 设置二进制模式成功 |
| 213 | 文件大小或时间信息 | `SIZE`、`MDTM` |
| 226 | 数据连接关闭，传输完成 | 文件下载或上传完成 |
| 250 | 文件操作成功 | 删除、重命名、切换目录 |
| 257 | 当前路径或目录创建信息 | `PWD`、`MKD` |
| 350 | 请求文件操作继续 | `REST` 或 `RNFR` |
| 421 | 服务不可用 | 服务器关闭连接 |
| 530 | 登录失败 | 用户名或密码错误 |
| 550 | 文件不可用 | 文件不存在、权限不足 |

FTP 还支持多行响应。例如 `FEAT` 命令会返回服务器支持的扩展能力：

```text
211-Features supported:
 EPSV
 MDTM
 MLST type*;perm*;size*;modify*;
 REST STREAM
 SIZE
 UTF8
211 End FEAT.
```

本项目对单行响应和多行响应均进行了支持。多行响应的结束规则是：最后一行以相同三位状态码和空格开头。

### 2.4 主动模式与被动模式

FTP 数据连接有主动模式和被动模式两种常见工作方式。

主动模式中，客户端通过 `PORT` 命令告诉服务器自己的 IP 和端口，服务器主动连接客户端。由于现代客户端常处在 NAT、校园网、防火墙或 WSL/虚拟化环境中，外部服务器很难主动连接客户端，因此主动模式容易失败。

被动模式中，客户端发送 `PASV` 或 `EPSV` 命令，请求服务器打开一个临时数据端口。服务器返回该端口后，由客户端主动连接服务器的数据端口。被动模式更适合现代网络环境，也是 WinSCP、FileZilla 等客户端的常用默认模式。

本项目默认使用被动模式，优先尝试 `EPSV`，失败后回退到 `PASV`：

- `EPSV` 返回扩展被动模式端口，适用于 IPv4/IPv6。
- `PASV` 返回服务器 IP 和端口，兼容传统 FTP 服务器。

### 2.5 二进制传输与文件完整性

FTP 支持 ASCII 模式和二进制模式。ASCII 模式可能对换行符进行转换，适合早期文本传输；二进制模式不修改文件字节流，更适合通用文件传输。

本项目在登录成功后立即发送：

```text
TYPE I
```

其中 `I` 表示 Image，即二进制模式。这样可以保证图片、压缩包、文档、可执行文件和文本文件都按原始字节传输，避免因换行转换导致文件损坏。

### 2.6 目录列表与文件元数据解析

远端文件系统可视化依赖目录列表命令。FTP 服务器常见目录列表方式包括：

- `MLSD`：机器可解析格式，字段更稳定，例如类型、大小、修改时间、权限等。
- `LIST`：传统目录列表，格式可能因 Unix、Windows、服务器实现不同而不同。

本项目连接后通过 `FEAT` 获取服务器能力，若服务器支持 `MLSD/MLST`，优先使用 `MLSD`；否则回退到 `LIST`。解析后统一转换为 `FtpEntry` 数据结构，供界面层展示。

远端列表中的每个项目包含：

- 名称
- 类型：文件、目录、链接、未知
- 大小
- 修改时间
- 权限
- 原始列表行
- 扩展事实字段

### 2.7 断点续传原理

断点续传是本项目的重要功能之一。其核心思想是：传输中断后，不重新传输已有部分，而是从已完成的字节偏移继续。

下载续传过程：

1. 检查本地是否已有同名部分文件。
2. 获取本地文件大小，作为偏移量 `offset`。
3. 通过 `SIZE remote_path` 获取远端完整大小。
4. 若本地大小小于等于远端大小，则发送 `REST offset`。
5. 服务器返回 `350` 后，发送 `RETR remote_path`。
6. 客户端以追加模式写入本地文件。

上传续传过程：

1. 获取本地文件总大小。
2. 通过 `SIZE remote_path` 获取远端已有文件大小。
3. 若远端大小小于等于本地大小，则发送 `REST remote_size`。
4. 服务器返回 `350` 后，发送 `STOR remote_path`。
5. 客户端从本地文件对应偏移处继续读取并发送。
6. 若服务器拒绝 `REST`，则回退为完整覆盖上传并在日志中提示。

断点续传依赖 FTP 的 `REST` 命令，以及服务器对 `REST STREAM` 的支持。本项目会从 `FEAT` 响应中记录服务器能力，并在实际传输时根据服务器响应进行兼容处理。

### 2.8 并发、阻塞与图形界面响应

Socket 网络通信可能发生阻塞，例如连接超时、目录列表等待服务器响应、文件传输耗时较长。如果直接在 GUI 主线程中执行网络操作，界面会卡顿甚至无响应。

本项目采用后台线程处理网络任务：

- 连接服务器在后台线程执行。
- 远端目录刷新在后台线程执行。
- 上传和下载由 `TransferManager` 的 worker 线程按队列执行。
- UI 更新通过 Qt 的 `Signal/Slot` 机制回到主线程。

这种设计既保证了网络操作的完整性，也避免了界面冻结。

### 2.9 文件传输队列与任务状态机

文件传输可能有多个任务同时排队。为了统一管理传输，本项目抽象出 `TransferTask` 和 `TransferManager`。

任务状态包括：

- `queued`：排队中
- `running`：传输中
- `paused`：已暂停
- `completed`：已完成
- `failed`：失败
- `cancelled`：已取消

每个任务记录：

- 任务 ID
- 上传或下载方向
- 本地路径
- 远端路径
- 是否启用续传
- 已传输字节数
- 总字节数
- 实时速度
- 错误信息

暂停和取消通过 `threading.Event` 实现。传输循环每次读写一个数据块后检查事件状态，若暂停则抛出内部 `TransferInterrupted("paused")`，若取消则抛出 `TransferInterrupted("cancelled")`。继续时重新入队，并通过断点续传从已有进度继续。

### 2.10 协议日志的意义

本项目提供协议日志面板，实时展示客户端命令和服务器响应，例如：

```text
C USER demo
S 331 Username ok, send password.
C PASS ********
S 230 Login successful.
C TYPE I
S 200 Type set to: Binary.
C EPSV
S 229 Entering extended passive mode
C RETR requirements.txt
S 150 File status okay. About to open data connection.
S 226 Transfer complete.
```

协议日志有两个重要作用：

1. 证明项目确实从 socket 层亲自实现 FTP 协议交互，而不是调用高级 FTP 客户端库。
2. 便于调试网络问题，例如认证失败、数据连接失败、权限不足、服务器不支持续传等。

### 3.4 模块设计

#### 3.4.1 协议层 FtpSession

`FtpSession` 是系统最核心的底层协议类，负责：

- 建立和关闭控制连接。
- 发送 FTP 命令。
- 解析 FTP 单行和多行响应。
- 维护当前工作目录。
- 获取服务器扩展能力。
- 建立被动模式数据连接。
- 执行目录列表、上传、下载和文件管理命令。

主要接口包括：

```python
connect(host, port, username, password, timeout, passive_mode)
list(path="")
cwd(path)
cdup()
pwd()
size(remote_path)
download_file(remote_path, local_path, resume=True)
upload_file(local_path, remote_path, resume=True)
delete(remote_path)
rmdir(remote_path)
mkdir(remote_path)
rename(old_path, new_path)
close()
```

控制连接由一个 socket 维护。数据连接每次按需通过 `EPSV/PASV` 创建，用完即关闭。

#### 3.4.2 解析层 parsers

解析层负责将服务器返回的协议文本转换为结构化数据，主要包括：

- 多行 FTP 响应解析。
- `EPSV` 响应解析。
- `PASV` 响应解析。
- Unix 风格 `LIST` 目录解析。
- Windows 风格 `LIST` 目录解析。
- `MLSD` 机器可读目录解析。

解析结果统一封装为 `FtpEntry`，避免 UI 层直接处理不同服务器返回格式。

#### 3.4.3 传输层 TransferManager

`TransferManager` 负责上传和下载任务队列。界面层只需要调用：

```python
enqueue_download(remote_path, local_path, resume=True)
enqueue_upload(local_path, remote_path, resume=True)
pause_task(task_id)
resume_task(task_id)
cancel_task(task_id)
```

传输层每次执行任务时都会建立独立 FTP 会话。这样做的优点是：

- 文件传输不会阻塞主控制连接。
- 上传下载失败不会破坏目录浏览会话。
- 后台传输可以独立重试。
- 任务状态更容易管理。

#### 3.4.4 存储层 SiteStore

站点管理使用 SQLite 保存连接信息。每个站点包含：

- 站点名称
- 主机
- 端口
- 用户名
- 是否保存密码
- 密码
- 更新时间

系统默认不保存密码。只有用户勾选“保存密码”时才写入密码字段。

为了避免每次连接后重复保存站点，系统通过 `host + port + username` 建立唯一索引。同一服务器和用户再次保存时会更新已有站点，而不是插入重复记录。

#### 3.4.5 界面层 MainWindow

界面采用类似 WinSCP Commander 的双栏布局：

- 左侧：本地文件系统。
- 右侧：远端 FTP 文件系统。
- 顶部：高频工具栏。
- 底部：传输队列和协议日志。
- 独立窗口：站点管理。

界面设计原则是：日常文件操作集中在双栏文件管理区，连接配置放在独立站点管理窗口，避免所有信息堆在同一界面中。

### 3.5 连接管理功能设计

连接管理包括站点管理、连接、断开连接、连接状态显示和错误提示。

#### 3.5.1 站点管理

用户点击“站点管理”后，可以配置：

- 站点名
- 主机
- 端口
- 用户名
- 密码
- 是否保存密码

保存后站点显示在左侧列表中。双击站点可以回填连接信息。

#### 3.5.2 连接流程

连接流程如下：

1. 从站点管理窗口读取主机、端口、用户名、密码。
2. 后台线程创建 `FtpSession`。
3. 调用 `socket.create_connection()` 建立控制连接。
4. 读取服务器 `220` 欢迎响应。
5. 发送 `USER`。
6. 若服务器返回 `331`，继续发送 `PASS`。
7. 登录成功后发送 `TYPE I`。
8. 发送 `FEAT` 获取服务器能力。
9. 发送 `PWD` 获取当前目录。
10. 刷新远端目录列表。
11. 更新界面状态为“已连接”。

#### 3.5.3 断开连接

断开连接时，客户端尝试发送：

```text
QUIT
```

随后关闭 reader 和控制 socket。若服务器已断开，客户端会捕获异常并强制释放资源。

### 3.6 文件系统可视化设计

本地和远端均采用表格方式显示文件系统。

左侧本地面板包含：

- 名称
- 大小
- 类型
- 修改时间

右侧远端面板包含：

- 名称
- 类型
- 大小
- 修改时间
- 权限

每个目录列表顶部都加入 `..` 行，表示返回父目录。这种设计与 Windows 文件管理器、WinSCP、Total Commander 等软件一致，便于用户快速理解。

目录打开方式：

- 双击目录进入。
- 选中 `..` 返回父目录。
- 点击“上级”按钮返回父目录。
- 在远端路径输入框中输入路径后点击“打开”。

### 3.7 文件管理功能设计

本项目支持以下文件系统管理操作：

| 功能 | 本地 | 远端 |
| --- | --- | --- |
| 打开目录 | 支持 | 支持 |
| 打开文件 | 调用系统默认应用 | 临时下载后调用系统默认应用 |
| 新建目录 | 支持 | `MKD` |
| 重命名 | 本地 rename | `RNFR/RNTO` |
| 删除文件 | 本地 unlink | `DELE` |
| 删除目录 | 本地 rmdir | `RMD` |
| 属性查看 | 路径、大小、时间 | 类型、大小、时间、权限、原始列表行 |
| 复制 | 支持 | 支持 |
| 剪切 | 支持 | 支持 |
| 粘贴 | 支持 | 支持 |

远端打开文件时，系统会先将文件下载到临时缓存目录，再调用操作系统默认程序打开。程序启动和关闭时会自动清理缓存目录，避免产生垃圾文件。

### 3.8 上传、下载与拖拽传输

上传方式：

- 选中左侧本地文件，点击“上传”。
- 右键本地文件，选择“上传到远端”。
- 从左侧拖拽文件到右侧远端面板。
- 本地文件复制后，在远端面板粘贴。

下载方式：

- 选中右侧远端文件，点击“下载”。
- 右键远端文件，选择“下载到左侧”。
- 从右侧拖拽文件到左侧本地面板。
- 远端文件复制后，在本地面板粘贴。

上传和下载都会进入传输队列。任务执行过程会实时更新：

- 状态
- 进度条
- 已传输大小
- 总大小
- 当前速度
- 错误信息

### 3.9 复制、剪切和粘贴设计

本项目实现了内部文件剪贴板，不依赖系统剪贴板文本内容。剪贴板记录：

- 来源面板：本地或远端
- 操作类型：复制或剪切
- 文件名
- 是否目录
- 本地路径或远端路径

支持快捷键：

- `Ctrl+C`：复制
- `Ctrl+X`：剪切
- `Ctrl+V`：粘贴

不同来源和目标的处理方式：

| 来源 | 目标 | 复制行为 | 剪切行为 |
| --- | --- | --- | --- |
| 本地 | 本地 | `copy2/copytree` | `shutil.move` |
| 本地 | 远端 | 上传 | 上传成功后删除本地源文件 |
| 远端 | 本地 | 下载 | 下载成功后删除远端源文件 |
| 远端 | 远端 | 临时下载再上传 | `RNFR/RNTO` 服务器端移动 |

由于标准 FTP 没有“服务器端复制文件”命令，因此远端到远端复制采用临时下载再上传的方式实现。远端到远端剪切可以使用 `RNFR/RNTO`，效率更高。

### 3.10 传输队列与断点续传设计

传输队列以表格方式展示每个任务。用户可以对任务执行：

- 暂停
- 继续 / 重试
- 取消

暂停并不会删除已经传输的部分文件。再次继续时，系统会根据本地或远端已有大小自动执行断点续传。

下载续传时：

- 读取本地部分文件大小。
- 发送 `REST offset`。
- 发送 `RETR remote_path`。
- 追加写入本地文件。

上传续传时：

- 发送 `SIZE remote_path` 获取远端已有大小。
- 发送 `REST offset`。
- 发送 `STOR remote_path`。
- 从本地文件 offset 位置继续发送。

### 3.11 协议日志设计

协议日志面板实时显示命令和响应：

- `C` 表示客户端发送命令。
- `S` 表示服务器返回响应。
- `!` 表示客户端内部提示或兼容处理。

密码在日志中会脱敏，例如：

- 连接超时

- 登录失败
- 服务器关闭控制连接
- 数据连接失败
- 权限不足
- 文件不存在
- 服务器不支持某些命令

本项目通过自定义异常类型区分：

## 4. 项目特色与总结

本项目的主要特色如下：

- 底层协议亲自实现：直接使用 Python `socket` 完成 FTP 控制连接和数据连接。
- 功能覆盖完整：上传、下载、断点续传、目录浏览、文件管理、站点管理、传输队列、协议日志均已实现。
- 交互体验完善：双栏文件管理、右键菜单、拖拽传输、复制剪切粘贴、快捷键操作接近专业 FTP 客户端。
- 可演示性强：协议日志能直观看到 FTP 命令和响应，本地 demo 服务器支持完整上传下载流程。
- 结构清晰：协议层、传输层、存储层、界面层职责分离，便于维护和扩展。
- 兼容性考虑充分：默认使用被动模式，支持 `EPSV/PASV`，支持 Windows 和 WSL 开发运行环境。

通过本实验，我们不仅实现了一个实用的图形化 FTP 客户端，也深入理解了应用层协议设计、TCP socket 编程、控制连接与数据连接分离、状态码协议解析、断点续传、并发任务队列和 GUI 与网络线程协作等计算机网络与软件工程核心知识。
